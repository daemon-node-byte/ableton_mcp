# Ableton Live 12 API Comparison and Codegen Prep

Date: 2026-04-09
Project: AbletonMCP_v2
Assumption mode: compare current remote script surface to documented/observed Ableton Live 12 API concepts and assume rough accuracy where they align

## Status update after the end-to-end slice

This repo now has:
- a real Python MCP server package under `mcp_server/`
- a canonical command registry in `mcp_server/command_specs.py`
- a thinner Remote Script entrypoint that delegates behavior into domain modules

Several previously suspicious contracts were also corrected against the official Live Object Model:
- `get_cpu_load` now reads `Application.average_process_usage`
- `get_session_path` now reads `Song.file_path`
- `get_arrangement_length` now reads `Song.song_length` with clip-end fallback
- `create_arrangement_audio_clip` now uses `Track.create_audio_clip(file_path, position)`
- `create_take_lane` now uses `Track.create_take_lane()`
- `create_midi_clip_in_lane` now uses `TakeLane.create_midi_clip(start_time, length)`
- `load_instrument_or_effect` now prefers `Track.insert_device(...)` for native devices and keeps browser URI loading as a provisional path

## Context

The current Remote Script was generated by a prior agent that timed out and was later abandoned. Because of that, this codebase should be treated as partially generated and potentially incomplete.

That means:
- the command dispatcher may advertise features whose implementations are partial or missing
- naming may be more complete than behavior
- some methods may be stubs, incorrect, or inconsistent
- the right near-term goal is code generation with strong guardrails, not blind trust in current code

## High-level comparison

The current `AbletonMCP_Remote_Script/__init__.py` exposes a large command surface that broadly matches the kinds of objects and capabilities visible in the Ableton Live 12 API / Live Object Model ecosystem:

### Areas that align well with the Live 12 object model

#### Song / transport
The remote script exposes commands for:
- tempo
- time signature
- play / stop / continue
- recording states
- metronome
- arrangement loop
- locators / cues
- arrangement time navigation
- automation re-enable
- undo / redo

This is consistent with the `Song` and `Song.View` style capabilities visible in Live's API ecosystem.

#### Tracks and mixer
The remote script exposes commands for:
- track creation and deletion
- duplicate track
- track name / color
- volume / pan / mute / solo / arm
- monitoring / delay
- routing input / output
- return tracks and send levels
- master track controls

This aligns with `Track`, `MixerDevice`, and routing-related control concepts that are commonly exposed through Live integrations.

#### Session clips
The remote script exposes commands for:
- create / delete / duplicate clip
- clip naming and color
- fire / stop clip
- get / set / remove notes
- loop and markers
- gain / pitch / warp mode
- quantize / duplicate loop
- clip automation read / write / clear

This is strongly aligned with `Clip`, `ClipSlot`, and the session-view workflow known to be supported by Live-adjacent APIs.

#### Arrangement clips
The remote script exposes commands for:
- get arrangement clips
- create arrangement MIDI clip
- create arrangement audio clip
- delete / resize / move arrangement clip
- add notes to arrangement clip
- get arrangement clip notes
- duplicate to arrangement

This aligns with the fact that the Live 12 ecosystem clearly includes Arrangement View concepts, clips in arrangement lanes, cue points, and timeline-oriented editing.
It also aligns with evidence from other tools like `ableton-copilot-mcp` that arrangement operations are feasible.

#### Devices, racks, chains, plugin-facing controls
The remote script exposes commands for:
- get track devices
- get device parameters
- set device parameter
- parameter access by name
- toggle / enable / delete / move device
- show / hide plugin window
- load instrument or effect
- get device class name
- select device
- rack chains
- rack macros
- chain devices
- chain mute / solo / volume
- drum rack pad operations

This aligns with the Live API object families around:
- `Device`
- `PluginDevice`
- `RackDevice`
- `Chain`
- `DeviceParameter`
- `DrumPad`

#### Browser and take lanes
The remote script exposes commands for:
- browser tree queries
- browser path queries
- browser search
- drum kit loading
- take lane queries and operations

This is directionally consistent with Live 12 concepts and manual-visible features, especially take lanes in Live 12.
Browser support is the most implementation-sensitive area, but it is still a reasonable target for a custom Remote Script.

## Comparison result

### Conclusion
At the level of command taxonomy, the current remote script is not obviously overreaching.
Most of what it claims to expose is compatible with known Live 12 concepts and with the broader Live Object Model / Remote Script ecosystem.

That is good news.

### Important caveat
This does **not** prove correctness of implementation.
It only means the command surface is believable enough to use as a code generation baseline.

So we should proceed under this working assumption:

- the current dispatcher is a plausible target capability map
- individual implementations may still be partial, broken, or missing
- code generation should preserve the current capability map where it is reasonable, while tightening organization, contracts, and completion status

## Likely weak spots to treat carefully

Even if the capability names are believable, these areas are the most likely to be incomplete or fragile:

1. `create_arrangement_audio_clip`
   - likely depends on exact file import semantics
   - likely platform/path sensitive

2. browser loading methods
   - custom browser integration often fails at the implementation level even when the API concept is valid

3. take lane commands
   - Live 12 feature area, but often less documented in third-party examples
   - higher chance of partial generation

4. plugin window control
   - `show_plugin_window` / `hide_plugin_window` may be more implementation-sensitive than parameter access

5. automation methods
   - clip automation is believable, but exact envelope handling can be tricky

6. nested rack and chain traversal
   - believable and valuable, but easy to implement incorrectly if object traversal is sloppy

## What this means for the next code generation pass

We should **not** regenerate the project from scratch blindly.
We should instead use the current codebase as a partial scaffold and impose structure around it.

## Recommended code generation strategy

### 1. Preserve the command surface first
Treat the current dispatcher as the source list of intended capabilities.
Before deleting or rewriting behavior, preserve a canonical command catalog.

### 2. Split the monolithic remote script
The current `AbletonMCP_Remote_Script/__init__.py` is too large for safe iteration.
The next codegen pass should move logic into modules like:

- `song_ops.py`
- `track_ops.py`
- `session_clip_ops.py`
- `arrangement_ops.py`
- `device_ops.py`
- `rack_ops.py`
- `browser_ops.py`
- `take_lane_ops.py`
- `view_ops.py`
- `utils.py`

The `__init__.py` should become a thin bootstrap plus command dispatcher.

### 3. Add completion markers
For each command implementation, classify it as one of:
- confirmed
- likely-complete
- partial
- stub
- unverified

This will help codegen avoid pretending uncertain code is production-ready.

### 4. Add strict command contracts
Every command should have:
- required params
- optional params
- expected result schema
- expected error conditions

This matters a lot for MCP tool generation later.

### 5. Keep plugin support pragmatic
For third-party plugins like Serum 2:
- treat parameter enumeration as the base primitive
- do not promise deep plugin UI scripting
- design aliases and profiles on top of exposed parameters only

## Summary

The current remote script looks directionally compatible with what Ableton Live 12 appears able to expose.
That is enough to proceed toward code generation.

But we should proceed with the assumption that the implementation is partial and possibly inconsistent because it came from an abandoned timed-out agent.

So the right move now is:
- preserve the current capability map
- create repo guidance for future agents
- prepare for structured refactoring and completion, not speculative redesign
